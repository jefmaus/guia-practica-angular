Glosario de Conceptos Clave

Los decoradores, conceptos y patrones más usados en Angular moderno

@C
@Component
Decorador de componente
Core
Convierte una clase TypeScript en un componente Angular. Desde Angular 17 standalone: true es el valor por defecto.
💡Usa siempre ChangeDetectionStrategy.OnPush y Signals para máximo rendimiento.
Componente moderno TS
@Component({
  selector: 'app-usuario',
  standalone: true,
  imports: [NgOptimizedImage, DatePipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <h2>{{ nombre() }}</h2>
    <p>{{ creado | date:'dd/MM/yyyy' }}</p>
  `
})
export class UsuarioComponent {
  nombre = signal('Ana García');
  creado = new Date();
}
@I
@Injectable / Service
Lógica reutilizable
DI
Marca una clase como inyectable. Con providedIn: 'root' crea un singleton global. Usa inject() en vez del constructor.
💡Los servicios son el lugar correcto para lógica compartida, HTTP y estado global.
Service moderno TS
@Injectable({ providedIn: 'root' })
export class UsuarioService {
  private http = inject(HttpClient);
  private url  = '/api/usuarios';

  getAll() {
    return this.http.get<Usuario[]>(this.url);
  }
}
IO
@Input / @Output
Comunicación padre-hijo
Core
@Input() recibe datos del padre. @Output() emite eventos. En Angular 17+ existen input() y output() basados en Signals.
💡Regla: los datos bajan con @Input, los eventos suben con @Output.
Signal Inputs/Outputs (v17+) TS
export class CardComponent {
  titulo    = input.required<string>();
  subtitulo = input('Sin subtítulo');
  seleccionado = output<number>();

  onSelect(id: number) {
    this.seleccionado.emit(id);
  }
}

inject()
DI funcional moderno
v14+
Reemplaza al constructor con parámetros. Solo puede usarse en contexto de inyección. Permite crear funciones reutilizables con DI.
💡inject() es la forma preferida. Más limpio, composable y funciona en guards y helpers.
Guard funcional TS
export const authGuard: CanActivateFn = () => {
  const auth   = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn()
    ? true
    : router.createUrlTree(['/login']);
};
|P
Pipes
Transformadores de template
Core
Transforman valores con el operador |. Las pure pipes solo recalculan cuando cambia la referencia. async gestiona subscripciones automáticamente.
Pipe custom + built-ins TS
{{ fecha | date:'dd/MM/yyyy HH:mm' }}
{{ precio | currency:'COP':'symbol':'1.0-0' }}
{{ texto | uppercase | slice:0:50 }}

@Pipe({ name: 'truncar', standalone: true })
export class TruncarPipe {
  transform(val: string, len = 50): string {
    return val?.length > len
      ? val.slice(0, len) + '…' : val;
  }
}

OP
ChangeDetection OnPush
Performance crítica
Perf
Default revisa toda la app ante cualquier evento. OnPush solo revisa cuando cambia una referencia de @Input o un Signal emite.
💡Siempre usa OnPush. Con Signals + OnPush → Zoneless, Angular solo actualiza los nodos que cambian.

  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class OptimizadoComponent {
  items  = signal<Item[]>([]);
  total  = computed(() =>
    this.items().reduce((s, i) => s + i.precio, 0)
  );
}

Signals — Estado Reactivo Moderno

El sistema de reactividad de Angular 16+. Más simple y eficiente que RxJS para estado local.

signal(valor)
Estado mutable. Se lee invocándolo. Se actualiza con .set(), .update() o .mutate().
const contador = signal(0);
contador();                    // leer → 0
contador.set(5);              // asignar
contador.update(n => n + 1); // transformar
computed(fn)
Valor derivado. Solo recalcula cuando sus dependencias cambian. De solo lectura.
const precio = signal(100);
const iva    = signal(0.19);
const total = computed(
  () => precio() * (1 + iva())
); // → 119
effect(fn)
Side effect que corre al cambiar sus signals. Para sync, logging o localStorage.
const tema = signal('dark');
effect(() => {
  document.body.dataset.theme = tema();
  localStorage.setItem('tema', tema());
});
toSignal(obs$)
Convierte un Observable RxJS en Signal. Gestiona la suscripción automáticamente.
import { toSignal } from '@angular/core/rxjs-interop';

export class Lista {
  datos = toSignal(inject(Svc).getAll(),
    { initialValue: [] });
}
🏪 Signal Store — gestión de estado sin NgRx clásico
@Injectable({ providedIn: 'root' })
export class CarritoStore {
  private _items = signal<Producto[]>([]);

  readonly items    = this._items.asReadonly();
  readonly total    = computed(() => this._items().reduce((s, p) => s + p.precio, 0));
  readonly cantidad = computed(() => this._items().length);
  readonly isEmpty  = computed(() => this._items().length === 0);

  agregar(p: Producto) { this._items.update(items => [...items, p]); }
  eliminar(id: number) { this._items.update(items => items.filter(p => p.id !== id)); }
  vaciar()              { this._items.set([]); }
}

Signal Forms
Nueva API de formularios
Experimental
Nueva API de formularios basada en Signals lanzada en Angular 21. El modelo es un signal sincronizado automáticamente con el template. Combina la simplicidad de template-driven con el poder de reactive forms.
⚠️ Experimental — la API puede cambiar antes de estabilizarse.
Signal Forms en acción TS
import { form } from '@angular/forms';

@Component({ template: `
  <input [field]="loginForm.email()" />
  <input [field]="loginForm.password()"
         type="password" />
  <p>{{ user().email }}</p>
`})
export class LoginComponent {
  user = signal({ email: '', password: '' });
  loginForm = form(this.user); // sincronización automática
}
LS
linkedSignal()
Signal con reset automático
v18+
Signal writable que se resincroniza automáticamente cuando su fuente cambia. Perfecto para selecciones que deben resetear al cambiar la lista fuente.
💡Caso clásico: ítem seleccionado que se resetea al cambiar la lista o categoría.
Selector que se resetea TS
const options = signal(['A', 'B', 'C']);

// Se resetea a options()[0] cuando options cambia
const selected = linkedSignal(() => options()[0]);

selected.set('B');           // escritura manual OK
options.set(['X', 'Y']);   // selected → 'X' auto
HR
httpResource()
Fetching reactivo con Signals
Experimental
Reemplaza el patrón ngOnInit + subscribe. Se re-fetcha automáticamente cuando los signals de sus parámetros cambian. Auto-cancela requests anteriores.
idle Sin parámetros válidos aún
loading Request en vuelo · cancela la anterior
resolved resource.value() disponible
error resource.error() disponible
Ejemplo reactivo TS
const page = signal(1);
const q    = signal('');
const products = httpResource(() => ({
  url: '/api/products',
  params: { page: page(), q: q() }
}), { defaultValue: [] });

products.value()   // los datos
products.status()  // 'idle'|'loading'|...'
products.error()   // el error si falló
Z
Zoneless Angular
Sin Zone.js — default v21
Default v21
Desde Angular 21 los proyectos nuevos no incluyen Zone.js. Angular detecta cambios solo cuando un Signal cambia. Bundles más pequeños, CD más preciso y predecible.
✅ Default en proyectos nuevos v21. Activa el provider para migrar existentes.
Migrar a Zoneless TS
// app.config.ts
providers: [
  provideExperimentalZonelessChangeDetection()
  // + elimina zone.js de polyfills array
]

export class RegistroComponent {
  private fb  = inject(FormBuilder);
  private svc = inject(AuthService);

  form = this.fb.group({
    nombre:   ['', [Validators.required, Validators.minLength(2)]],
    email:    ['', [Validators.required, Validators.email]],
    password: ['', [Validators.required, Validators.minLength(8), passwordFuerteValidator]],

  onSubmit() {
    if (this.form.invalid) { this.form.markAllAsTouched(); return; }
    this.svc.register(this.form.getRawValue()).subscribe();
  }
}

const passwordFuerteValidator: ValidatorFn = (c) =>
  /[A-Z]/.test(c.value) && /[0-9]/.test(c.value)
    ? null : { passwordDebil: true };

HttpClient — Peticiones HTTP

CRUD, interceptores funcionales y el nuevo httpResource()

HTTP
Service con CRUD + interceptor
@Injectable({ providedIn: 'root' })
export class ProductoService {
  private http = inject(HttpClient);
  private url  = 'https://api.ejemplo.com/productos';

  getAll() {
    return this.http.get<Producto[]>(this.url).pipe(
      shareReplay(1), catchError(this.handleError)
    );
  }
  create(data: Omit<Producto, 'id'>) {
    return this.http.post<Producto>(this.url, data);
  }
  update(id: number, data: Partial<Producto>) {
    return this.http.patch<Producto>(`${this.url}/${id}`, data);
  }
  delete(id: number) {
    return this.http.delete<void>(`${this.url}/${id}`);
  }
}

// Interceptor funcional (moderno)
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).getToken();
  if (!token) return next(req);
  return next(req.clone({
    setHeaders: { Authorization: `Bearer ${token}` }
  }));
};

🔍 Buscador con debounce + switchMap
export class BuscadorComponent {
  private svc = inject(ProductoService);
  termino   = signal('');
  cargando  = signal(false);

  resultados = toSignal(
    toObservable(this.termino).pipe(
      debounceTime(300),
      distinctUntilChanged(),
      filter(q => q.trim().length >= 2),
      tap(() => this.cargando.set(true)),
      switchMap(q => this.svc.buscar(q).pipe(
        catchError(() => of([]))
      )),
      tap(() => this.cargando.set(false))
    ),
    { initialValue: [] }
  );
}

Lifecycle Hooks

Ciclo de vida completo de un componente Angular

1
constructor()
Solo inyección de dependencias con inject(). Nunca accedas al DOM ni hagas HTTP aquí. Los @Input no están disponibles todavía.
2
ngOnChanges(changes)
Antes de ngOnInit y ante cada cambio de @Input. Recibe SimpleChanges con valores anteriores y nuevos.
3
ngOnInit()
✅ El más usado. Una sola vez tras inicializar @Inputs. Aquí va la lógica de arranque y peticiones HTTP iniciales.
4
ngDoCheck()
Cada ciclo de detección de cambios. Muy costoso. Solo para detección manual en casos excepcionales.
5
ngAfterContentInit()
Tras proyectar contenido con ng-content. Los @ContentChild están disponibles. Se ejecuta una vez.
6
ngAfterViewInit()
✅ Tras renderizar la vista y componentes hijos. @ViewChild disponible. Interactúa con el DOM aquí.
7
ngAfterViewChecked()
Tras cada verificación de vista. Muy costoso. Puede causar ExpressionChangedAfterItHasBeenChecked.
X
ngOnDestroy()
✅ Antes de destruir. Desuscribe Observables, limpia timers. Usa DestroyRef / takeUntilDestroyed() en código moderno.

SSR, SSG & Hydration

Server-Side Rendering, Static Site Generation y Partial Hydration

✅ Usa SSR cuando...
App pública / e-commerce / blog
SEO es crítico para el negocio
Core Web Vitals son KPI
Primer render rápido importa
Redes lentas / dispositivos móviles
Landing pages / marketing
❌ No necesitas SSR si...
Backoffice / admin / dashboard
Detrás de login (no indexable)
Datos 100% dinámicos por usuario
Equipo pequeño (+ complejidad)
Herramienta interna / B2B SaaS
App de análisis de datos
🌍 Activar SSR con Hydration completa
ng new mi-app --ssr  // genera con SSR

// app.config.ts
provideClientHydration(withHttpTransferCache()),
// withHttpTransferCache: evita doble fetch cliente/servidor

// Partial hydration con @defer (lo mejor de ambos mundos)
@defer (on viewport) {
  <heavy-chart /> // NO hidrata hasta entrar al viewport
}
@defer (on idle) {
  <footer-heavy /> // cuando el browser está ocioso
}
Regla híbrida Angular 21: SSR + @defer (on viewport) = primer HTML rápido del servidor + componentes pesados cargan lazy. Activa siempre withHttpTransferCache().

Change Detection — Evolución

De Zone.js + Default hasta Zoneless + Signals en Angular 21

Zone.js + Default
Zone intercepta TODOS los eventos (setTimeout, Promises, DOM events). CD en todo el árbol. 100 componentes = 100 revisiones. Bundle +40KB de zone.js.
Zone.js + OnPush
Solo revisa cuando @Input cambia por referencia, un Signal emite, o async pipe emite. Mucho más eficiente. El estándar recomendado antes de v21.
Default Angular 21
Zoneless + Signals
Sin Zone.js. CD solo cuando un Signal cambia. Bundle más pequeño. Solo los componentes afectados se actualizan. Más predecible y debuggeable. Default en proyectos nuevos v21.

DI Avanzado — Jerarquía

El árbol de injectors de Angular y cómo Angular resuelve dependencias

RT
Root Injector
providedIn: 'root' — Singleton global. Una instancia para toda la app.
PL
Platform Injector
Servicios del platform (PlatformRef). Raro usarlo directamente.
ENV
Environment Injector
Route-level. Nueva instancia por ruta lazy. Ideal para servicios de feature aislados.
CMP
Component Injector
providers:[] en @Component. Nueva instancia por componente y sus hijos.
CH
Child Component Injector
Hereda del padre. Si no encuentra el servicio, sube en la jerarquía hasta el Root.
Angular busca el servicio de hijo a padre. Si llega al Root sin encontrarlo → error de runtime. Usa inject(Service, {optional: true}) para servicios opcionales.

Comparativas VS

Cuándo usar qué — decisiones directas y al grano

Estado del componente
✅ Signals
Simple, sin subscribe. Reactivo en template. Zoneless-ready. computed memoizado. El futuro.
VS
RxJS BehaviorSubject
Verboso. Hay que gestionar unsubscribe. Potente para streams complejos.
Usar ahora: Signals para estado local/derivado. RxJS para HTTP, websockets y streams complejos encadenados.
Formularios en Angular 21
✅ Signal Forms (v21)
Experimental. Sin boilerplate. Full type-safe. Model = signal. Combina TD + RF.
VS
Reactive Forms
Estable y maduro. FormGroup/FormControl con TypedForms. Enterprise-ready.
Usar ahora: Signal Forms en proyectos nuevos que aceptan experimental. Reactive Forms en producción estable.
Carga de datos HTTP
✅ httpResource()
Reactivo a signals. Auto-cancela. Estados como signals. Sin subscribe. Experimental v21.
VS
HttpClient + subscribe
Clásico y estable. Control total con RxJS. Interceptores sin problemas.
Usar ahora: httpResource() para fetching reactivo. HttpClient directo para lógica RxJS avanzada o interceptores.
Estado global
✅ NgRx SignalStore
Poco boilerplate. Composable. Signals-based. Moderno. Proyectos nuevos.
VS
NgRx Clásico
Actions + Reducers + Effects. Más boilerplate. Muy predecible en equipos grandes.
Usar ahora: NgRx SignalStore en nuevos. Para estado simple: Signals en servicio con providedIn:'root'. NgRx clásico si ya existe.
Test runner
✅ Vitest (v21 default)
Rápido. Fake timers. Snapshots. Paralelización. DX moderna.
VS
Karma / Jasmine
Legacy. Más lento. Sigue soportado pero no es el default en v21.
Usar ahora: Vitest en proyectos nuevos. No hay urgencia de migrar proyectos Karma existentes.
Change Detection
✅ Zoneless + Signals
Default v21. Sin Zone.js. Bundle más pequeño. CD solo cuando Signal cambia.
VS
Zone.js + OnPush
Clásico. Paso intermedio de migración hacia zoneless.
Usar ahora: Zoneless en proyectos nuevos v21. OnPush + Signals en proyectos existentes antes de migrar.
Renderizado
SSR + Hydration
HTML en servidor. SEO óptimo. FCP rápido. Requiere servidor Node.
VS
CSR (SPA clásica)
Simple, solo estático. SEO limitado. Ideal para apps internas.

Componentes accesibles

Sin estilos. Tu design system. Signals-based. Accesibilidad correcta desde el inicio.

Material Design listo. Estable. No lo reemplaza Angular Aria — son complementarios.
Usar ahora: Angular Aria para design systems custom. Angular Material para Material Design listo. Son complementarios.
